# ETL-пайплайн для потоковой обработки документов

Система автоматически обрабатывает русскоязычные бизнес-документы: пользователь загружает PDF / DOCX / XLSX / CSV / TXT через веб-интерфейс, система разбирает файл, определяет тип документа (счёт-фактура, акт, договор, ТОРГ-12, УПД, платёжное поручение), извлекает реквизиты (контрагенты с ролями, ИНН / КПП, даты, суммы) с помощью локальной языковой модели, проверяет валидность, сохраняет в базу и предоставляет интерфейс для поиска, аудита и ручной корректировки.

## Стек

| Слой | Технология | Назначение |
|------|-----------|------------|
| Очередь сообщений | Apache Kafka | Развязывает приём документов и их обработку |
| Обработчик потока | aiokafka consumer (Python) | Читает из очереди, прогоняет документ через стадии извлечения |
| База данных | PostgreSQL 16 | Хранит документы, реквизиты, аудит-трейл, обеспечивает полнотекстовый поиск |
| Файловое хранилище | Локальный Docker volume | Исходные blob'ы, имя файла = sha256-хеш |
| API-сервер | FastAPI + asyncpg | REST для UI: документы, поиск, авторизация, ручная корректировка |
| Веб-интерфейс | Vue 3 + Vuetify | SPA с темой (light / dark), серверной пагинацией, drag-and-drop загрузкой |
| Авторизация | JWT (15 мин) + refresh-токен в Redis (30 дней) | Stateless access, мгновенный logout, защита от кражи токена |
| Языковая модель — основная | Qwen3.5-9B (квантизация 4-bit) через MLX | Классификация типа документа + извлечение реквизитов с ролями + сводка одним промптом |
| Языковая модель — альтернативная | Sber GigaChat (Lite / Pro / Max) | Облачный бэкенд, переключается одной env-переменной |
| OCR для PDF-сканов | PaddleOCR-VL 1.5 | Распознавание текста, таблиц и layout на сканах без текстового слоя |
| Парсинг текстовых форматов | pdfplumber, python-docx, openpyxl, pandas | Покрывают 99 % digital-документов |
| Мониторинг | Prometheus + Grafana | Дашборд с throughput, latency, ошибками, очередью Kafka |
| Логирование | Loki + Promtail | Агрегация логов всех сервисов, корреляция с метриками в Grafana |
| Деплой | docker-compose | Запуск всего стека одной командой |

## Структура репозитория

```

├── services/
│   ├── extract/             # приём → парсинг → OCR-fallback → отправка в Kafka
│   ├── transform/           # чтение из Kafka → извлечение реквизитов → сохранение
│   ├── api/                 # REST для UI: авторизация, документы, поиск, админ
│   └── ui/                  # Vue 3 SPA + nginx
├── tools/

│   └── llm_bench/           # инструмент сравнительного тестирования языковых моделей
├── samples/real/            # реальные русскоязычные PDF + ground-truth разметка
└── docs/
    └── llm-comparison.md    # результаты нагрузочного тестирования
```

## Запуск

```bash
cp .env.example .env       # один раз: создать локальный конфиг
make up                    # поднять весь стек
```

Файл `.env` локальный, в `.gitignore`. Для облачного бэкенда GigaChat в нём указывается `GIGACHAT_AUTH_KEY` (Authorization Key из личного кабинета на developers.sber.ru).

Локальные языковые модели запускаются нативно на хосте (Apple Silicon Metal через MLX, в Docker не помещаются по архитектурным причинам):

```bash
make ai-up                 # поднять Qwen3.5-9B и PaddleOCR-VL
```

Сервисы и порты:

| Сервис | URL | Доступ |
|--------|-----|--------|
| Веб-интерфейс | http://localhost:5173 | `admin` / `admin` |
| REST API (Swagger) | http://localhost:8000/docs | JWT из формы логина |
| Grafana | http://localhost:3001 | `admin` / `admin` |
| Prometheus | http://localhost:9090 | — |
| PostgreSQL | `localhost:5432` | `etl` / `etl` |

Полная справка по командам Makefile — в комментариях [Makefile](./Makefile):

```bash
make up             # поднять стек (идемпотентно)
make down           # остановить контейнеры (данные сохраняются)
make logs           # хвост логов всех сервисов
make psql           # консоль PostgreSQL
make rebuild        # пересборка без кеша
make clean          # удалить все данные (документы, БД, метрики, логи)
make ai-up          # запустить локальные языковые модели на хосте
make bench          # сравнительный тест языковых моделей
```

## Поток данных

1. **Загрузка** — пользователь добавляет файл через интерфейс (drag-and-drop поддерживает папки) или REST API. Сервис приёма определяет тип файла, считает sha256, проверяет дубликаты (повторная загрузка возвращает существующий документ с пометкой), сохраняет файл и публикует событие в очередь Kafka.

2. **Извлечение реквизитов** — обработчик читает событие из Kafka и прогоняет документ через несколько стадий:
   - **Регулярные выражения** находят все возможные кандидаты для ИНН (с проверкой контрольной суммы), КПП, дат, сумм и контрагентов с организационно-правовыми формами.
   - **Языковая модель** получает текст документа и имя файла, возвращает структурированный ответ: тип документа, контрагенты с ролями (продавец, покупатель, плательщик), ключевые даты, итоговая сумма и НДС, краткая сводка.
   - **Сверка** сравнивает ответ модели с regex-кандидатами и отмечает галлюцинации (значения, отсутствующие в исходном тексте).
   - **Валидация** проверяет форматы и нормализует данные (даты → ISO, суммы → Decimal).
   - **Сохранение** в PostgreSQL с записью полного аудит-трейла каждой стадии.

3. **Просмотр и корректировка** — на странице документа пользователь видит извлечённые реквизиты, исходный файл, аудит обработки. При необходимости можно вручную исправить любое поле или контрагента — система логирует, кто и что изменил. Кнопка «Перепроцессить» отправляет документ на обработку заново. Для подробной сводки можно дополнительно запустить генерацию через языковую модель в режиме streaming.

4. **Поиск** — полнотекстовый поиск по содержимому документа и реквизитам реализован через встроенные средства PostgreSQL с поддержкой русского языка и ранжированием результатов.

## Гарантии обработки

- **Дедупликация по содержимому**: повторная загрузка одного и того же файла не создаёт дубль — система возвращает существующий документ с пометкой и текущим статусом.
- **Однократная обработка**: pipeline спроектирован так, что повторная обработка одного и того же события из Kafka даёт тот же результат (idempotent UPSERT по идентификатору документа).
- **Восстановление после сбоя**: незавершённые документы продолжают обработку при перезапуске сервиса (Kafka хранит позиции, БД — статусы).


## Тестовые документы

В разделе «Админ» можно сгенерировать синтетические документы для проверки системы — поддерживаются 6 типов в 4 форматах (PDF / DOCX / XLSX / CSV) с реалистичными реквизитами (фейковые ИНН с правильной контрольной суммой, корректные даты и суммы). Поддержка комбинаций «тип × формат» имитирует реальный документооборот: акты и договоры — Word, ТОРГ-12 / УПД — Excel, счёт-фактура — все форматы.

После генерации система автоматически загружает файлы в pipeline и показывает результат обработки по каждому: успешно загружен / с предупреждением / с ошибкой / дубликат.

Дополнительно генератор умеет создавать **edge-кейсы** (документы с ИП в ИНН-12, отсутствующими реквизитами, многострочными названиями, нестандартным порядком блоков) и **junk-документы** (текстовые не-документы, пустые файлы, битые PDF, файлы с неправильным расширением) — для проверки устойчивости pipeline и срабатывания правила отбраковки нерелевантных документов.
